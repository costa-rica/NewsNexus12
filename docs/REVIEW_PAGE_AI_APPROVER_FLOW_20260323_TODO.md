### phase 1 decisions

1. new API content endpoint

- Route: `GET /analysis/ai-approver/review-article-content/:articleId`
- Auth: required via existing API auth middleware
- Purpose: dedicated review-page modal endpoint for article title and read-only content display
- Expected success shape:

```json
{
  "result": true,
  "articleId": 123,
  "title": "Article title",
  "hasArticleContent": true,
  "content": "Canonical ArticleContents02 content",
  "contentSource": "article-contents-02"
}
```

- Defensive no-content shape:

```json
{
  "result": true,
  "articleId": 123,
  "title": "Article title",
  "hasArticleContent": false,
  "content": null,
  "contentSource": null
}
```

- Expected behavior:
  - read `ArticleContents02` directly for modal display
  - return `hasArticleContent = true` only when at least one `ArticleContents02` row exists for the article
  - do not modify or reuse an existing content endpoint for this feature

2. new API one-off queue endpoint

- Route: `POST /analysis/ai-approver/review-page/start-job`
- Auth: required via existing API auth middleware
- Request body:

```json
{
  "articleId": 123,
  "name": "Prompt name",
  "promptInMarkdown": "User-written or copied prompt",
  "sourcePromptVersionId": 45
}
```

- Request notes:
  - `sourcePromptVersionId` is optional and used for UI traceability only if needed later
  - the API creates the prompt row before calling worker-python
  - the API, not the portal, generates `description`
  - the API always stores the prompt as inactive

- Prompt row created by API:

```json
{
  "name": "Prompt name",
  "description": "userId:7, articleId:123, date:2026-03-23",
  "promptInMarkdown": "User-written or copied prompt",
  "isActive": false,
  "endedAt": null
}
```

- Expected success shape:

```json
{
  "result": true,
  "jobId": "queue-job-id",
  "status": "queued",
  "endpointName": "/ai-approver/review-page/start-job",
  "promptVersionId": 456,
  "articleId": 123
}
```

3. new worker-python route

- Route: `POST /ai-approver/review-page/start-job`
- Purpose: queue a single-article single-prompt AI approver run
- Request body:

```json
{
  "articleId": 123,
  "promptVersionId": 456
}
```

- Expected success shape:

```json
{
  "jobId": "queue-job-id",
  "status": "queued",
  "endpointName": "/ai-approver/review-page/start-job"
}
```

- Expected behavior:
  - load exactly one prompt row by `promptVersionId`
  - load exactly one article by `articleId`
  - use canonical `ArticleContents02` content when present
  - fall back to `Articles.description` only when `ArticleContents02` content is not found
  - call OpenAI for that single prompt only
  - insert one `AiApproverArticleScores` row for that prompt/article attempt

4. review table and modal implications

- The bottom review table should receive a boolean field indicating whether `ArticleContents02` exists for the row’s article.
- The new content icon should be rendered only when that boolean is true.
- The article `id` sort must remain unchanged, so the icon must not change the existing `id` accessor sort behavior.
- The one-off score row should flow into the existing top-score query and existing `AI Approver Details` modal automatically once written, with no separate score-display model.
